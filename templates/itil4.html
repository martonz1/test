Key Terms
Service
A means of enabling value co-creation without the customer managing costs/risks directly
Value
Perceived benefits relative to costs. Always defined by the customer, not the provider
Outcome vs Output
Output = the thing delivered. Outcome = the result it enables. ITIL cares about outcomes
Utility
"Fit for purpose" - does it do what it's supposed to do?
Warranty
"Fit for use" - is it available, reliable, secure enough?
CI/CD vs ITIL
Change management = ITIL's way to ship safely without breaking live services
7 Guiding Principles - always apply, regardless of context
Focus on value Every action must trace back to value for someone
Start where you are Don't reinvent. Assess what exists first, build on it
Progress iteratively + feedback Small steps, frequent feedback loops. Don't big-bang
Collaborate + promote visibility Work together, share info. Hidden work = hidden risk
Think & work holistically No silo thinking. Every part affects the whole system
Keep it simple & practical If it doesn't add value, remove it. Complexity is waste
Optimize & automate Optimize the process first, then automate it. Don't automate chaos
Key Practices (14 of 34 - the ones you'll use daily)
Incident Management
Service broken restore it fast. Log, triage, fix. e.g. server down at 3am
Problem Management
Find root cause so incident never recurs. e.g. why does the server keep crashing?
Change Enablement
Control how changes go to production. Avoid breaking live services
Service Desk
Single point of contact. Captures all requests + incidents from users
Service Request Mgmt
Handle standard user requests. e.g. new laptop, account access
Event Management
Monitor and respond to system events. e.g. planned downtime, alerts
IT Asset Management
Track all hardware/software assets. Know what you own and where
Relationship Management
Manage relationships with customers, users, stakeholders
Supplier Management
Govern third-party providers, contracts, performance
Info Security Mgmt
Protect confidentiality, integrity, availability of data
Continual Improvement
Structured approach to improving services and practices over time
Monitoring & Event Mgmt
Observe services systematically. Detect issues before users do
Incident vs Problem - most confused pair
Incident

Unplanned interruption. Fix it now. Speed matters more than root cause.

"Login is broken - restore it"
Problem

Unknown root cause of one or more incidents. Investigate properly.

"Why does login keep breaking?"
A Problem can exist without an active Incident (proactive problem management). A known workaround = Known Error.
End-to-end value flow example
  • User need: "I need a dev environment" (demand)
  • Plan: what env, what stack, what SLA?
  • Engage: dev team, security, cloud supplier aligned
  • Design: spec the environment, security rules
  • Obtain/Build: spin up cloud VMs, configure CI/CD
  • Deliver & Support: hand over, document, monitor
  • Value delivered: dev can build and ship code (outcome)
  • Improve: next time, automate provisioning
// Foundation Exam - what they test
  • Know all 7 guiding principles by name
  • Know all 4 dimensions - apply them to scenarios
  • Know 6 SVC activities + what Improve means
  • Distinguish incident vs problem vs request
  • Understand utility + warranty - both needed for value
  • Know difference: output vs outcome
  • 34 practices exist - know ~15 well for the exam
  • SVS components: principles, governance, SVC, practices, improvement
  • 40 questions, 65% to pass, 60 min, closed book
Tip: exam scenarios always have a "most correct" answer. Map every option back to: does this create value?
